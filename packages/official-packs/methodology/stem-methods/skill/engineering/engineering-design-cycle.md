# Engineering design cycle

The **engineering design process** (NGSS ETS, CDIO Conceive–Design–Implement–
Operate, design thinking variants) iterates **define → ideate → prototype → test →
refine** against requirements and constraints.

## When to use

- MechE, EE, systems, UX-in-engineering, intro design courses
- Any unit where trade-offs, specs, and failure analysis are learning goals
- Professionals audience often pairs Merrill *problem-first* with this *cycle*

## Unit arc

| Phase | Learner does | Block families |
|---|---|---|
| **Define** | Requirements, constraints, success metrics | **Narrative** + **check** (prioritize requirements) |
| **Ideate** | Multiple concepts; down-select with rationale | **Spatial** sketch family, **compare**, **narrative** |
| **Prototype** | Build minimum viable model/sim/code | **Manipulative**, **executable**, **spatial** |
| **Test** | Measure vs specs; failure analysis | **Data**, **check** (numeric tolerances) |
| **Refine** | Iterate; document trade-offs | Loop manipulative/executable/data; **formalism** for design log |

## Rules

- At least **two ideation alternatives** before committing — avoids first-idea lock-in.
- Test phase must compare to **quantified** requirements (`data` + `check` families).
- Document *why* a design was rejected, not only the winner.

## Anti-patterns

- Single-pass "design" with no test against requirements.
- Prototype that's only cosmetic UI with no measurable behavior.
