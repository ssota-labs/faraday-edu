# Problem-based learning (PBL)

**Problem-based learning** starts from an **ill-structured authentic problem**;
learners identify what they need to learn, study resources, apply knowledge, and
reflect. Common in medical and early engineering education (Barrows; systematic
reviews in engineering show gains in problem-solving and self-directed learning).

## When to use

- Freshman engineering, integrated STEM, professional case opens
- When the *problem* should drive the learning agenda, not a chapter list
- Pairs with `audience` Merrill for professionals; this file is the **unit arc**

## Unit arc

| Phase | Learner does | Block families |
|---|---|---|
| **Problem presentation** | Engage with messy scenario; list unknowns | **Narrative** (rich scenario), **check** (what do we need to know?) |
| **Idea generation** | Hypotheses, learning issues,分工 | **Narrative** + **check** (prioritize issues) |
| **Self-study / resources** | Acquire concepts (may span nodes) | **Formalism**, **data**, **executable** as needed — *just-in-time* |
| **Apply & test** | Solution attempt on the original problem | **Manipulative**, **executable**, **check** (mission) |
| **Debrief** | What worked; what would we do differently | **Narrative** + **check** (reflection) |

## Rules

- Problem stays **ill-structured** early — resist front-loading all formalism.
- Learning issues the learner generates should map to later nodes (curriculum
  alignment with `lecture-design` backward design).
- Apply phase uses **realistic constraints** (units, budgets, failure modes).

## Anti-patterns

- Fake PBL: problem is a word problem with one taught trick.
- Self-study that's a video with no application back to the case.
