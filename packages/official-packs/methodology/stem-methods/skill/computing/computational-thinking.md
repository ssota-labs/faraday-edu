# Computational thinking (CT)

**Computational thinking** (Wing 2006; CSTA/ISTE frameworks) emphasizes
**decomposition, abstraction, pattern recognition, and algorithm design** —
often before or alongside a specific programming language. Unplugged and plugged
activities both count.

## When to use

- CS0, K–12 CS, data literacy modules, "thinking like a computer scientist"
- Math/ science units on algorithms (sorting, search, simulation design)
- When syntax is not the primary learning goal

## Unit arc

| Phase | Learner does | Block families |
|---|---|---|
| **Decompose** | Break task into subproblems | **Narrative** scenario + **spatial** (flow/map) or **check** ordering |
| **Abstract** | Hide detail; name patterns | **Manipulative** (simplified model), **formalism** (light) |
| **Pattern** | Recognize repetition / structure | **Data** or **manipulative** across cases |
| **Algorithm** | Specify steps; trace execution | **Executable** (trace) or **check** (trace MCQ) |
| **Evaluate** | Efficiency, correctness, edge cases | **Check** + **executable** tests |

## Rules

- Tracing beats reading code for novices — use **check** (what's the value at step
  k?) before free coding.
- Abstraction level must match audience: children → manipulative/unplugged;
  undergrad → executable.
- Edge cases are explicit learning goals, not footnotes.

## Anti-patterns

- CT = "learn Python syntax" with no decomposition practice.
- Algorithms with no traceable execution path for the learner.
