# Move 4 — Keep it fair

- **No answer leaks.** During the exam, don't render the correct answer, the
  `hint`, or a running "correct so far" signal. If a `tutor` is present, its
  no-leak guard must hold (see the `tutor` pack) — the tutor clarifies wording, it
  does not answer items.
- **Randomize sensibly.** Shuffle option order and, where you have a bank, sample
  items per section — but keep each learner's exam blueprint-equivalent (same
  outcomes × weights). Don't randomize difficulty into an unfair draw.
- **Timing is accessible.** If timed, state the limit up front, warn before expiry,
  and don't hard-fail on a lost connection mid-answer. Prefer generous or optional
  timing unless the construct *is* speed.
- **One attempt vs. practice.** Be explicit which this is. Practice tests should
  allow retries and show the review immediately; a graded mock should submit once,
  then review.

→ **Faraday:** these are authoring rules, not a runtime lock — it's the learner's
own app. The point is an exam the *creator* can trust to measure what they meant.
